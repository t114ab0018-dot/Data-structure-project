# AGENTS.md

## Project Overview
本專案是「校園租屋與包租代管媒合平台」的前端展示 Demo。
目標是做出一個可以在瀏覽器中打開的靜態網站，展示：
- 依據生活費試算租屋預算與對應房源
- 視覺化通勤時間與周邊生活機能的地圖節點
- 租金補貼物件篩選機制與白話文租屋知識庫
- 房東對租客的初步篩選介面與公益出租人稅金試算
- 房東/業者的多房源集中管理與修繕進度儀表板
- 本專案主要用於課堂展示與期末報告，不是實際營運的租屋系統。

## Tech Stack
請只使用：
- HTML
- CSS
- JavaScript
- 不使用後端
- 不使用資料庫
- 不使用登入系統
- 不使用 npm / React / Vue 等框架
最終作品必須可以直接用 `index.html` 開啟。

## File Structure
請維持以下檔案結構：
├── index.html
├── style.css
├── script.js
├── data/    # 任何其他資料都放置於此
│   ├── houseData.js       # 房源與行情假資料
│   ├── landlordData.js    # 房東與稅務假資料
├── AGENTS.md
├── PRD.md
├── USER_STORY.md
└── README.md

## Must Have
- 租屋預算試算與補貼篩選介面
- 結合地圖節點的通勤時間與機能評估
- 房東管理儀表板（收租、修繕進度、稅金試算）
- 產權驗證標章及校方警示推播介面展示
- 使用 Hash Map 或 Graph 等資料結構管理房源與地理節點的說明
- 適合簡報展示的畫面
-學生身分驗證介面
-房東房產驗證介面

## Do Not Do
- 不要做太複雜的系統
- 不要加入真實地圖 API（如 Google Maps 真實串接，可用靜態節點圖取代）
- 不要重寫全部程式
- 不要做需要安裝才能跑的功能

## 驗收標準 (重要)

### 1. 基本執行
- 使用者可以直接打開 `index.html` 看到完整畫面
- 不需要安裝額外套件
- 不需要登入
- 不需要後端伺服器
- 不需要 API key
- 頁面載入後不能出現明顯錯誤畫面
- 瀏覽器 console 不應該有會影響功能的 JavaScript error

### 2. Demo 流程
使用者必須能完成一個完整展示流程：
1. 進入首頁，選擇身分（學生租客 / 房東 ）。
2. 選擇或看到一個明確的情境（例如：學生尋找補貼房源、房東試算稅金）。
3. 操作主要功能（試算預算、篩選標章、查看修繕進度）。
4. 系統產生明確結果（顯示符合的房源列表、稅金減免結果、發布推播成功）。
5. 結果可以被截圖或口頭說明。
6. 整個流程可以在 3 分鐘內展示完。
如果是互動式作品，至少要有一條「保證可成功展示」的 demo path

### 3. 核心功能
- 至少完成 MVP 中定義的核心功能（預算與補貼篩選、房東稅務與修繕管理、產權驗證）。
- 核心功能不能只是靜態文字描述，必須有可操作或可觀察的結果。
- 每個按鈕或主要互動元件都應該有明確作用。
- 不應存在點了沒有反應、但看起來像主要功能的按鈕。
- 如果功能尚未完成，請不要放在主要畫面上誤導使用者。

完成時必須能回答：
- 使用了哪一種資料結構？
- 資料是怎麼存的？
- 為什麼這裡適合用這種資料結構？
- 這個資料結構如何影響功能運作？

### 4. 檔案與程式碼
- 主要檔案命名清楚，例如 `index.html`, `style.css`, `script.js`。
- 不要把大量 JavaScript 全部塞在 HTML 裡。
- 不要留下大量沒用到的程式碼。
- 不要留下測試用假按鈕、假頁面或無意義文字。
- 函式名稱應該能看出用途。
- 如果有資料檔，資料結構要清楚，不要全部寫成難以閱讀的一大坨字串。

### 5. 穩定性
- 重新整理頁面後仍可以正常使用。
- 操作順序稍微不同時，不應直接壞掉。
- 沒有選資料就按按鈕時，系統應該給出提示，而不是報錯。
- 如果輸入錯誤，畫面應該有清楚回饋。
- Demo 前應至少完整跑過 3 次主要流程。

### 6. 範圍控制
以下內容不是必要項目，不應該影響 MVP 完成：
- 登入系統
- 後端資料庫
- 真實金流
- 真實通知系統
- 真實 GPS
- 複雜 AI 模型
- 大量頁面
- 過度精緻但無法操作的動畫

### 10. 最終完成定義
當以下條件都達成時，才算完成：
- 可以正常開啟作品。
- 可以完成主要 demo 流程。
- 有清楚的輸入、處理、輸出。
- 至少一個資料結構有被實際用在功能中。
- 畫面適合展示。
- 團隊能說明作品邏輯。
- 沒有明顯會讓 demo 中斷的錯誤。

補充 : 以上驗收過程建議使用 browser use 功能 (/browser) 做測試
