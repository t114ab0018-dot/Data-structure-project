# AGENTS.md

## Project Overview
本專案是「校園租屋與包租代管媒合平台」的前端展示 Demo。
目標是做出一個可以在瀏覽器中打開的靜態網站，展示我們為 3 分鐘 Demo 設計的核心流程：
- 首頁三方入口（學生、房東、系統管理員）
- 學生端：校方 Portal 單一入口 SSO 登入模擬、視覺化地圖找房與條件篩選、白話文租屋知識庫、室友媒合系統、租屋生活面板（合約與報修追蹤）
- 房東端：防偽與產權審查模擬、待辦事項儀表板（收租/修繕進度）、內政部公版合約線上發送
- 平台端：身分與產權驗證中心、數位合約稽核與糾紛調解數位足跡面板
- 本專案主要用於課堂展示與期末報告，不是實際營運的租屋系統。

## Tech Stack
請只使用：
- HTML
- CSS
- JavaScript
- 不使用後端
- 不使用資料庫
- 不使用真實登入系統
- 不使用 npm / React / Vue 等框架
最終作品必須可以直接用 `index.html` 開啟。

## File Structure
請維持以下檔案結構：
├── index.html
├── style.css
├── script.js
├── data/    # 任何其他資料都放置於此
│   ├── mockData.js        # 包含假資料（如：北科大測試帳號、房東清單、合約狀態）
├── AGENTS.md
├── PRD.md
├── USER_STORY.md
└── README.md

## Must Have
- 清晰的首頁與三種身分（學生、房東、平台）切換動線
- 學生端：地圖找房篩選 UI、知識庫 Tooltip 提示、室友媒合問卷與結果（如：列出高契合度室友）、報修進度條展示
- 房東端：待辦事項中心（顯示欠租與修繕）、物件狀態管理、一鍵發送公版合約按鈕
- 平台端：待審核房東清單（身分證與權狀核對介面）、黑名單紅燈警告、糾紛調解面板
- 使用適合的資料結構（如 Hash Map 或 Array 處理篩選邏輯與室友契合度計算）
- 適合簡報展示的乾淨畫面與直覺的 UI 互動

## Do Not Do
- 不要做太複雜的系統
- 不要加入真實地圖 API（如 Google Maps 真實串接，可用靜態地圖圖片搭配絕對座標的節點取代）
- 不要重寫全部程式
- 不要做需要安裝才能跑的功能
- 不要實作原本已作廢的功能（如：預算試算器、稅金試算、過往租客評價）

## 驗收標準 (重要)

### 1. 基本執行
- 使用者可以直接打開 `index.html` 看到完整畫面
- 不需要安裝額外套件
- 不需要登入
- 不需要後端伺服器
- 不需要 API key
- 頁面載入後不能出現明顯錯誤畫面
- 瀏覽器 console 不應該有會影響功能的 JavaScript error

### 2. Demo 流程
使用者必須能完成以下三條核心展示路徑之一（3 分鐘內可走完）：
1. **學生找房與合租情境**：進入首頁 -> 點擊「我是學生」-> 模擬 Portal 登入 -> 使用地圖篩選房源 / 填寫室友問卷並顯示配對結果 -> 查看報修進度。
2. **房東管理情境**：進入首頁 -> 點擊「我是房東」-> 模擬上傳產權 -> 進入待辦事項中心 -> 點擊發送內政部公版合約 / 更新修繕狀態。
3. **管理員稽核情境**：進入首頁 -> 點擊右上角「系統管理」-> 審核房東上傳的權狀 -> 檢視糾紛調解面板（展開雙方數位足跡）。

### 3. 核心功能
- 至少完成 MVP 中定義的核心功能（地圖篩選、室友配對、房東儀表板、後台審核）。
- 核心功能不能只是靜態文字描述，必須有可操作或可觀察的結果（例如點擊送出問卷後，畫面要真的切換到配對結果）。
- 每個按鈕或主要互動元件都應該有明確作用。
- 不應存在點了沒有反應、但看起來像主要功能的按鈕。
- 如果功能尚未完成，請不要放在主要畫面上誤導使用者。

完成時必須能回答：
- 使用了哪一種資料結構？
- 資料是怎麼存的？
- 為什麼這裡適合用這種資料結構？
- 這個資料結構如何影響功能運作？

### 4. 檔案與程式碼
- 主要檔案命名清楚，例如 `index.html`, `style.css`, `script.js`。
- 不要把大量 JavaScript 全部塞在 HTML 裡。
- 不要留下大量沒用到的程式碼。
- 不要留下測試用假按鈕、假頁面或無意義文字。
- 函式名稱應該能看出用途。
- 如果有資料檔，資料結構要清楚，不要全部寫成難以閱讀的一大坨字串。

### 5. 穩定性
- 重新整理頁面後仍可以正常使用。
- 操作順序稍微不同時，不應直接壞掉。
- 沒有選資料就按按鈕時，系統應該給出提示，而不是報錯。
- 如果輸入錯誤，畫面應該有清楚回饋。
- Demo 前應至少完整跑過 3 次主要流程。

### 6. 範圍控制
以下內容不是必要項目，不應該影響 MVP 完成：
- 真實資料庫
- 真實金流
- 真實通知系統
- 真實 GPS
- 複雜 AI 模型
- 大量頁面
- 過度精緻但無法操作的動畫

### 7. 最終完成定義
當以下條件都達成時，才算完成：
- 可以正常開啟作品。
- 可以完成主要 demo 流程。
- 有清楚的輸入、處理、輸出。
- 至少一個資料結構有被實際用在功能中。
- 畫面適合展示。
- 團隊能說明作品邏輯。
- 沒有明顯會讓 demo 中斷的錯誤。

補充：以上驗收過程建議使用 browser use 功能 (/browser) 做測試。
