# AGENTS.md

## Project Overview

本專案是「北科安心租 (Next-Gen Rental Platform)」的前端展示 Demo，專為前端 UI 生成工具量身打造。
目標是做出一個可以在瀏覽器中打開的靜態網站，精準還原指定的七大核心分頁結構與互動邏輯：

* **全域導覽**：固定式 Navbar（含品牌 Icon 與系統管理入口）。
* **首頁入口**：長滾動設計，包含主視覺 Hero Section、3 欄網格特色區、橫向滾動的 5 張 AI 推薦房源卡片，以及可展開的房東見證 Accordion。
* **學生端流程**：學號/密碼表單登入模擬、左右分割的地圖找房介面（Split View）、10 題生活習慣問卷與 3 張室友配對卡片動態展示、滿版閱讀器模式的合約簽署頁，以及 3 欄網格排列的租屋生活儀表板。
* **房東端流程**：包含 Dropzone 文件上傳與審核狀態提示的登入註冊面板、4 張頂部數據統計卡與直向排列的物件狀態列表。
* 本專案主要用於展示 UI 架構與前端頁面間的跳轉邏輯，不是實際串接後端的營運系統。

## Tech Stack

請只使用：

* HTML
* CSS
* JavaScript
* 不使用後端
* 不使用資料庫
* 不使用真實登入系統
* 不使用 npm / React / Vue 等框架
最終作品必須可以直接用 `index.html` 開啟。

## Must Have

* **全域 Navbar**：左側「北科安心租」Logo，右側「系統管理」連結，固定於畫面上方。
* **頁面 0 (首頁)**：兩個主要 CTA 按鈕必須能正確觸發跳轉至【第一分頁】與【第二分頁】。具備橫向滾動軸 (Carousel) 的 AI 推薦卡片，圖片需疊加「租補」Badge。
* **頁面 1 & 2 (登入驗證)**：置中表單面板。頁面 1 點擊登入後跳轉【第三分頁】；頁面 2 需有模擬上傳的 Dropzone 區塊與「等待審核」提示，並跳轉【第七分頁】。
* **頁面 3 (地圖找房)**：左側條件面板與右側靜態地圖搭配 Markers。必須實作兩種互動邏輯：點擊「套房」條件觸發跳轉【第四分頁】，點擊「房源卡片/簽約」跳轉【第五分頁】。
* **頁面 4 (室友配對)**：點擊「送出」問卷後，必須透過 JavaScript 動態顯示下半部的 3 張配對結果卡片（含契合度進度條）。
* **頁面 5 & 6 (簽約與學生後台)**：滿版合約閱讀器，底部 Sticky Bar 勾選後跳轉至學生儀表板，展示 3 張狀態網格卡片（租金狀態、報修追蹤、租約倒數）。
* **頁面 7 (房東後台)**：頂部 4 張數據卡片（須帶有 Icon 與趨勢提示），下方膠囊狀導覽列（我的物件需呈現 Active 狀態），以及直向排列的物件卡片（需帶有綠色/灰色狀態標籤）。
* 使用適合的資料結構（如 Array of Objects）來管理與渲染房源清單及配對卡片。

## Do Not Do

* 不要做太複雜的系統與多餘的分頁（嚴格遵守頁面 0 至頁面 7 的定義，不需實作真實的「系統管理」後台頁面，僅在 Navbar 保留入口）。
* 不要加入真實地圖 API（可用靜態地圖圖片搭配絕對座標的 CSS 標記點取代 Map Markers）。
* 不要重寫全部程式。
* 不要做需要安裝才能跑的功能。
* 不得偏離 UI 規格（不可隨意新增架構中未提及的側邊欄或浮動視窗）。

## 驗收標準 (重要)

### 1. 基本執行

* 使用者可以直接打開 `index.html` 看到完整首頁畫面
* 不需要安裝額外套件
* 不需要登入
* 不需要後端伺服器
* 不需要 API key
* 頁面載入後不能出現明顯錯誤畫面
* 瀏覽器 console 不應該有會影響功能的 JavaScript error

### 2. Demo 流程

使用者必須能流暢完成以下三條基於 UI 架構設計的核心展示路徑：

1. **學生找房與合約情境**：首頁 (頁面 0) -> 點擊「開始找房」 -> 學生登入 (頁面 1) -> 進入地圖房源查詢 (頁面 3) -> 點擊房源卡片的「簽約」 -> 進入合約簽署 (頁面 5) -> 點擊確認跳轉至我的租屋生活 (頁面 6) 查看租約倒數。
2. **學生找室友情境**：進入地圖房源查詢 (頁面 3) -> 篩選條件選擇「套房」觸發跳轉 -> 室友配對系統 (頁面 4) 填寫問卷 -> 點擊送出並於下方檢視 3 張配對結果卡片。
3. **房東管理情境**：首頁 (頁面 0) -> 點擊「我是房東」 -> 房東認證與登入 (頁面 2) -> 模擬上傳與審核跳轉 -> 進入房東管理後台 (頁面 7) 查看 4 張數據卡與物件出租狀態清單。

### 3. 核心功能

* 至少完成 MVP 中定義的畫面渲染與狀態跳轉（Carousel 滑動、Accordion 展開、Dropzone 視覺、進度條顯示）。
* 核心功能不能只是靜態文字描述，必須有可操作或可觀察的結果（例如頁面 4 點擊送出問卷後，畫面下方要真的渲染出 3 張卡片）。
* 每個按鈕或主要互動元件都應該有明確作用。
* 如果功能尚未完成，請不要放在主要畫面上誤導使用者。

完成時必須能回答：

* 使用了哪一種資料結構儲存頁面 3 的房源與頁面 7 的物件清單？
* 資料是怎麼存的？
* 這個資料結構如何配合 JavaScript 動態生成 DOM 元素（如卡片）？

### 4. 檔案與程式碼

* 主要檔案命名清楚，例如 `index.html`, `style.css`, `script.js`。
* 不要把大量 JavaScript 全部塞在 HTML 裡。
* 不要留下大量沒用到的程式碼。
* 函式名稱應該能看出用途（例如 `renderPropertyCards()`, `MapsToPage(pageId)`）。

### 5. 穩定性

* 重新整理頁面後仍可以正常使用。
* 操作順序稍微不同時，不應直接壞掉。
* 如果輸入錯誤，畫面應該有清楚回饋。
* Demo 前應至少完整跑過 3 次主要流程。

### 6. 範圍控制

以下內容不是必要項目，不應該影響 MVP 完成：

* 真實資料庫
* 真實金流
* 真實通知系統
* 真實 GPS 地圖串接
* 複雜 AI 模型運算（契合度請直接使用 mockData 中的假數值渲染）

### 7. 最終完成定義

當以下條件都達成時，才算完成：

* 可以正常開啟作品。
* 可以依照互動邏輯 1 與 2 完成主要 demo 跳轉流程。
* 嚴格遵守頁面 0 到 7 的 UI 版面配置與元件要求。
* 畫面適合展示，具備基本的 RWD 響應式或適當的寬度限制。
* 沒有明顯會讓 demo 中斷的錯誤。
